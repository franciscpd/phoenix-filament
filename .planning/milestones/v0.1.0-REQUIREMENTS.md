# Requirements: PhoenixFilament

**Defined:** 2026-03-31
**Core Value:** Developers can go from an Ecto schema to a fully functional, beautiful admin interface in minutes — with a declarative, idiomatic Elixir API that feels native to the Phoenix ecosystem.

## v1 Requirements

Requirements for v0.1.0 release. Each maps to roadmap phases.

### Foundation

- [x] **FOUND-01**: Mix project structured as a proper Hex package with correct supervision tree
- [x] **FOUND-02**: Core macro framework provides `use PhoenixFilament.Resource` with declarative DSL blocks (form, table)
- [x] **FOUND-03**: Field/column definitions are plain data structs evaluated at runtime (not compile-time macros)
- [x] **FOUND-04**: Macro delegation pattern prevents compile-time dependency cascades (thin delegation via `Macro.expand_literals/2`)

### Component Library

- [x] **COMP-01**: Text input component with label, placeholder, error display, and disabled state
- [x] **COMP-02**: Textarea component with configurable rows
- [x] **COMP-03**: Number input component with min/max/step support
- [x] **COMP-04**: Select component with static options and association-backed options
- [x] **COMP-05**: Checkbox/toggle component for boolean fields
- [x] **COMP-06**: Date and datetime picker components
- [x] **COMP-07**: Hidden field component
- [x] **COMP-08**: Button component with variants (primary, secondary, danger) and loading state
- [x] **COMP-09**: Badge component with color variants
- [x] **COMP-10**: Modal component using LiveView portals for proper z-index/overflow handling
- [x] **COMP-11**: Card component for content grouping
- [x] **COMP-12**: All components styled with Tailwind CSS v4 using CSS variable theming (no Tailwind class interpolation)

### Theming

- [x] **THEME-01**: Default theme with professional appearance (leveraging daisyUI 5 + Tailwind v4)
- [x] **THEME-02**: Theme customization via CSS variables (colors, fonts, spacing)
- [x] **THEME-03**: Light and dark mode support via CSS variables
- [x] **THEME-04**: Theme can be configured per-panel

### Form Builder

- [x] **FORM-01**: Declarative form definition via `form do ... end` DSL block
- [x] **FORM-02**: Form renders fields based on field type definitions (text_input, select, checkbox, etc.)
- [x] **FORM-03**: Full Ecto changeset integration — form submits through changeset, displays field-level errors inline
- [x] **FORM-04**: Support for separate create and update changeset functions
- [x] **FORM-05**: Form layout support (sections, columns/grid)
- [x] **FORM-06**: LiveView real-time validation on blur/change
- [x] **FORM-07**: Field visibility can be conditional (show/hide based on other field values)

### Table Builder

- [x] **TABLE-01**: Declarative table definition via `table do ... end` DSL block
- [x] **TABLE-02**: Configurable columns with label, formatting callback, and show/hide
- [x] **TABLE-03**: Server-side sorting (click column headers, default sort configurable)
- [x] **TABLE-04**: Server-side pagination with configurable page sizes (25/50/100)
- [x] **TABLE-05**: Global text search across configurable fields
- [x] **TABLE-06**: Empty state display when no records match
- [x] **TABLE-07**: Table uses LiveView streams for memory-efficient rendering
- [x] **TABLE-08**: Row actions (view, edit, delete) with confirmation for destructive actions
- [x] **TABLE-09**: Table filters (select filter, boolean filter, date range filter)
- [x] **TABLE-10**: Filter and pagination state preserved in URL params

### Resource

- [x] **RES-01**: `use PhoenixFilament.Resource` generates index, create, edit, and show pages from Ecto schema
- [x] **RES-02**: Resource auto-discovers schema fields and generates sensible defaults for form and table
- [x] **RES-03**: Resource allows full customization of form and table definitions via DSL
- [x] **RES-04**: Resource provides standard CRUD operations (list, get, create, update, delete) with Ecto repo integration
- [x] **RES-05**: Resource supports custom page titles, breadcrumbs, and navigation labels
- [x] **RES-06**: Delete action includes confirmation dialog

### Panel

- [x] **PANEL-01**: Admin panel shell with sidebar navigation, top bar, and content area
- [x] **PANEL-02**: Sidebar shows registered resources with icons and labels, with active state highlighting
- [x] **PANEL-03**: BYO authentication via configurable `on_mount` hook — panel checks auth, does not provide it
- [x] **PANEL-04**: Dashboard page as panel landing page with customizable layout
- [x] **PANEL-05**: Flash notifications for CRUD operations (success, error messages)
- [x] **PANEL-06**: Responsive layout — sidebar collapses on mobile
- [x] **PANEL-07**: Panel defined via router macro (`phoenix_filament_panel`) inside a scope
- [x] **PANEL-08**: Breadcrumb navigation on all pages

### Plugin Architecture

- [x] **PLUG-01**: Plugin behaviour with `register/2` and `boot/1` callbacks
- [x] **PLUG-02**: Plugin registration system per panel (each panel declares its plugins)
- [x] **PLUG-03**: Plugins resolved at runtime (on_mount), not compile-time — avoids recompilation cascades
- [x] **PLUG-04**: Built-in Resource system registered as a plugin (internals-as-plugins pattern)
- [x] **PLUG-05**: Plugin can register navigation items, routes, and components
- [x] **PLUG-06**: Plugin developer documentation with guide on creating community plugins

### Distribution

- [x] **DIST-01**: Published as Hex package (`phoenix_filament`)
- [x] **DIST-02**: `mix phx_filament.install` generator using Igniter for automated setup
- [x] **DIST-03**: Installer configures router, adds CSS/JS dependencies, creates initial admin panel module
- [x] **DIST-04**: Getting started documentation (installation, first resource, customization)

## Out of Scope

Explicitly excluded. Documented to prevent scope creep.

| Feature | Reason |
|---------|--------|
| Built-in authentication | Users bring their own auth — avoids ecosystem conflict |
| Dead view support | 100% LiveView — simplifies architecture, enables real-time features |
| Multi-tenancy | Extremely high complexity, defer to v0.3.0+ |
| Code generation (scaffold-style) | Loses framework upgrade benefits — framework approach preferred |
| Mobile native app | Web-first, Phoenix LiveView handles responsive |
| OAuth/social login | Auth is BYO, not our responsibility |
| Database-specific features | Must be database-agnostic (PostgreSQL, MySQL, SQLite) |
| Notification inbox UI | Complex feature, defer to v0.2.0+ |
