## Recommended Architecture

PhoenixFilament follows a layered architecture where a thin **Panel** layer orchestrates **Resource**, **Form**, and **Table** subsystems. Each layer has a clear single responsibility. The plugin contract is the seam between framework internals and community extensions.

```
┌─────────────────────────────────────────────────────────────┐
│  Panel  (shell, nav, routing, auth hook, plugin registry)   │
├────────────────┬──────────────┬──────────────────────────────┤
│   Resource     │   Pages      │  Plugin Registry             │
│ (CRUD wiring)  │ (LiveViews)  │  (behaviour + callbacks)     │
├────────────────┴──────────────┴──────────────────────────────┤
│            Form Builder        Table Builder                 │
│       (schema + fields)    (columns + filters + actions)     │
├─────────────────────────────────────────────────────────────┤
│                    Component Library                         │
│      (inputs, selects, modals, badges, buttons …)           │
├─────────────────────────────────────────────────────────────┤
│           Foundation  (Ecto integration, config, helpers)    │
└─────────────────────────────────────────────────────────────┘
```

Build order (bottom-up):

1. Foundation
2. Component Library
3. Form Builder
4. Table Builder
5. Resource abstraction
6. Panel + Plugin system
7. Mix generator (`mix phx_filament.install`)

## Component Boundaries

| Component | Responsibility | Communicates With |
|-----------|---------------|-------------------|
| **Foundation** | Ecto schema introspection, config reading, shared type helpers | Nothing (leaf layer) |
| **Component Library** | Stateless HEEx function components: inputs, buttons, badges, modals, layouts | Foundation (for type helpers) |
| **Form Builder** | Declares field schemas, resolves Ecto changesets, renders form HTML | Component Library (renders fields), Foundation (Ecto introspection) |
| **Table Builder** | Declares columns and filters, builds Ecto queries, paginates | Component Library (renders cells), Foundation (Ecto introspection) |
| **Resource** | Binds a Form + Table to an Ecto schema; generates CRUD LiveViews | Form Builder, Table Builder, Foundation |
| **Pages** | Concrete LiveView modules (Index, New, Edit, Show) produced by Resource | Resource (for config), Component Library (for layout shell) |
| **Panel** | Registers a route scope, sidebar nav, auth hook, and plugin list | Pages (routes them), Plugin Registry |
| **Plugin Registry** | Stores plugin modules; calls `register/1` and `boot/1` at mount time | Panel, Resource, Pages |
| **Mix Generator** | Writes config, imports JS/CSS, patches router | All (produces scaffolding) |

### Key Boundary Rules

- Form Builder and Table Builder must NOT know about Panel. They are standalone and composable.
- Resource is the only component that owns the relationship between Form and Table.
- Plugin Registry must NOT be invoked at compile time — it is a runtime concern resolved during panel mount.
- Component Library exposes only `Phoenix.Component` function components (no LiveViews). No side effects, no assigns beyond what is passed in.

## Data Flow

### Request Lifecycle (LiveView mount path)

```
Browser GET /admin/users
  → Phoenix Router scope (/admin, pipe_through :admin_browser)
  → live_session :admin_panel, on_mount: PhoenixFilament.AuthHook
    → AuthHook.on_mount/4  (calls user-supplied verify_plug, halts if unauthorized)
  → UsersLive.Index (generated by Resource macro)
    → mount/3:
        1. Resource.__resource__(:config) → fetch schema, repo, changeset fns
        2. Table.build_query(socket, params) → Ecto query
        3. Repo.all(query) → rows
        4. socket.assigns[:rows, :meta, :filters]
    → render/1:
        1. Panel shell component (<.admin_layout nav={...}>)
        2. Table component receives rows + column defs → renders HTML
```

### Form Submit Path

```
User submits form
  → handle_event("validate", params, socket)
      Form.changeset(schema, params, :validate) → changeset
      socket.assigns[:changeset]
  → handle_event("save", params, socket)
      Form.changeset(schema, params, :create) → changeset
      Repo.insert(changeset)
      → success: redirect to Index with flash
      → failure: socket.assigns[:changeset] with errors → re-render
```

### Plugin Registration Flow

```
Application start
  → Panel config evaluated (compile-time macro expansion)
  → Plugin modules listed in config

Panel LiveView mount
  → Plugin.register(panel_config) called for each plugin
      → plugin appends resources/pages to panel state
  → Plugin.boot(panel_config) called when panel is active
      → plugin performs runtime side effects (e.g., PubSub subscriptions)
```

## Macro DSL Design (Elixir Translation of FilamentPHP)

### Pattern: `__using__` + module attribute accumulation + `@before_compile`

This is the idiomatic Elixir pattern used by Ecto, Phoenix Router, and Absinthe.

```elixir
# User-facing API:
defmodule MyApp.Admin.UserResource do
  use PhoenixFilament.Resource,
    schema: MyApp.User,
    repo: MyApp.Repo

  @impl true
  def form(assigns) do
    [
      PhoenixFilament.Form.text_input(:name, label: "Full Name", required: true),
      PhoenixFilament.Form.email_input(:email, label: "Email Address"),
      PhoenixFilament.Form.select(:role, options: [:admin, :user])
    ]
  end

  @impl true
  def table(assigns) do
    [
      PhoenixFilament.Table.column(:name, label: "Name", sortable: true),
      PhoenixFilament.Table.column(:email, label: "Email"),
      PhoenixFilament.Table.column(:role, label: "Role")
    ]
  end
end
```

### How the macro compiles

`use PhoenixFilament.Resource` injects:
1. `@behaviour PhoenixFilament.Resource` — enforces `form/1` and `table/1` callbacks
2. Default implementations of optional callbacks (e.g., `filters/0 -> []`)
3. A `__resource__/1` function that carries config (schema, repo, changeset fns) for introspection at runtime
4. Route helper macros that the Panel can call to wire up LiveViews

`@before_compile` is used sparingly — only when the framework needs to inspect accumulated module attributes after all field declarations are resolved. Prefer explicit callback functions over implicit attribute accumulation to keep compile graphs small (HIGH confidence: see Elixir anti-patterns docs).

### Form Field Definition (data structure approach)

Fields are plain structs, not macros. This keeps them composable and testable:

```elixir
defmodule PhoenixFilament.Form.Field do
  defstruct [
    :name,        # atom — maps to Ecto field
    :type,        # :text | :email | :select | :checkbox | :textarea | ...
    :label,       # string
    :required,    # boolean
    :options,     # for selects: [{label, value}]
    :placeholder,
    :hidden_fn,   # fn(assigns) -> boolean — dynamic visibility
    :disabled_fn, # fn(assigns) -> boolean
    :default
  ]
end
```

The Resource callback `form/1` returns a list of `%Field{}` structs. The Form Builder renders them by iterating over the list and dispatching to the correct Component Library function component based on `:type`.

This separates _declaration_ (Resource) from _rendering_ (Form Builder + Component Library) — the same field list can be used for HTML rendering AND for driving Ecto changeset validation.

### Table Column Definition (same data structure approach)

```elixir
defmodule PhoenixFilament.Table.Column do
  defstruct [
    :name,        # atom — maps to Ecto field or virtual
    :label,       # string
    :sortable,    # boolean
    :searchable,  # boolean
    :format_fn,   # fn(value, row) -> rendered value
    :hidden_fn    # fn(assigns) -> boolean
  ]
end
```

## Plugin Architecture (Elixir Idioms)

### Behaviour Contract

```elixir
defmodule PhoenixFilament.Plugin do
  @callback id() :: atom()
  @callback register(panel :: map()) :: map()
  @callback boot(panel :: map()) :: :ok | {:error, term()}

  # Optional callbacks with default implementations
  @callback resources() :: [module()]
  @callback pages() :: [module()]
  @callback nav_items() :: [map()]

  defmacro __using__(_opts) do
    quote do
      @behaviour PhoenixFilament.Plugin
      def resources(), do: []
      def pages(), do: []
      def nav_items(), do: []
      defoverridable [resources: 0, pages: 0, nav_items: 0]
    end
  end
end
```

### Plugin Registration (Runtime, not Compile-Time)

Plugins are registered in Panel config and resolved at runtime mount — NOT at compile time. This avoids large recompilation graphs and allows per-panel customization:

```elixir
# In host app's config or panel module:
defmodule MyApp.AdminPanel do
  use PhoenixFilament.Panel,
    path: "/admin",
    plugins: [
      MyApp.Plugins.Blog,
      MyApp.Plugins.UserManagement
    ],
    auth_module: MyApp.AdminAuth
end
```

The Panel `on_mount` hook calls `Plugin.register/1` for each plugin, then calls `Plugin.boot/1`. This means plugin state is per-socket, not global — aligned with LiveView's process isolation model.

### Internals-as-Plugins

The built-in Resource system is itself registered as a plugin:

```elixir
defmodule PhoenixFilament.ResourcePlugin do
  use PhoenixFilament.Plugin

  def id(), do: :resource_plugin

  def register(panel) do
    resources = Application.get_env(:phoenix_filament, :resources, [])
    Map.put(panel, :resources, panel.resources ++ resources)
  end

  def boot(_panel), do: :ok
end
```

This guarantees the community plugin API is exactly the API the framework itself uses (HIGH confidence: this is the explicitly stated PROJECT.md requirement and matches FilamentPHP's design philosophy).

## Panel Routing Pattern

The Panel mounts as a router scope using `Phoenix.Router` macros. The host app calls a single macro:

```elixir
# In host app's router.ex:
import PhoenixFilament.Router

scope "/admin" do
  pipe_through [:browser, :fetch_current_user]

  phoenix_filament_panel MyApp.AdminPanel
end
```

Internally `phoenix_filament_panel/1` expands to:

```elixir
live_session :phoenix_filament_admin,
  on_mount: [PhoenixFilament.AuthHook, {MyApp.AdminPanel, :boot}],
  root_layout: {PhoenixFilament.Layouts, :admin} do

  live "/", PhoenixFilament.Pages.Dashboard, :index
  live "/resources/:resource", PhoenixFilament.Pages.Resource.Index, :index
  live "/resources/:resource/new", PhoenixFilament.Pages.Resource.New, :new
  live "/resources/:resource/:id/edit", PhoenixFilament.Pages.Resource.Edit, :edit
end
```

The resource identifier in the path resolves to a registered Resource module via a panel-level registry (a map stored in socket assigns during mount).

## Anti-Patterns to Avoid

### Anti-Pattern 1: Compile-Time Plugin Resolution
**What:** Resolving plugin list at compile time via `@before_compile`
**Why bad:** Any plugin change forces recompilation of the Panel module AND all its dependents. In large apps this triggers cascade recompilation.
**Instead:** Resolve plugins at runtime in `on_mount`. Store in socket assigns.

### Anti-Pattern 2: Global Process State for Panel Config
**What:** Storing panel state in a GenServer or ETS for all users to share
**Why bad:** LiveView is process-isolated by design. Shared mutable state creates race conditions and makes per-user panel customization (role-based nav) impossible.
**Instead:** Compute panel state per-socket in `on_mount`. Lightweight maps are cheap.

### Anti-Pattern 3: Macro-Heavy Field Declarations
**What:** Every field is a macro call that accumulates into a module attribute
**Why bad:** Macro accumulation creates compile-time dependencies; errors surface at compile time with confusing backtraces; testing requires full module compilation.
**Instead:** Fields are plain data (structs/maps) returned from `form/1` callback. Macros only for the outermost `use PhoenixFilament.Resource` boundary.

### Anti-Pattern 4: Duplicating Ecto Changeset Logic in the Framework
**What:** Framework infers validation from field definitions and generates changesets internally
**Why bad:** Elixir apps already have well-designed changeset functions. Duplicating this causes two sources of truth for validation.
**Instead:** Resource requires the user to supply `create_changeset/2` and `update_changeset/2` function references. Framework calls them; it does not generate them.

### Anti-Pattern 5: Deeply Nested LiveComponent Trees
**What:** Panel → Resource → Table → Row → Cell — each as a separate LiveComponent
**Why bad:** LiveComponents are stateful processes. Deep nesting multiplies message-passing overhead and makes parent-child coordination complex.
**Instead:** Panel shell is a LiveView. Table and Form are function components (stateless). Only the Panel-level LiveView holds state. LiveComponents are used only when isolated state genuinely benefits the UX (e.g., inline editing cell).

## Component Hierarchy (Runtime)

```
PhoenixFilament.Pages.Resource.Index  (LiveView — holds state)
├── PhoenixFilament.Components.AdminLayout  (function component — shell + nav)
│   ├── PhoenixFilament.Components.Sidebar  (function component)
│   │   └── nav items from panel assigns
│   └── <main slot>
│       └── PhoenixFilament.Table.Component  (function component)
│           ├── PhoenixFilament.Table.FilterBar  (function component)
│           ├── PhoenixFilament.Table.Header  (function component)
│           │   └── column headers (sortable, with sort indicators)
│           └── PhoenixFilament.Table.Row (function component, per row)
│               └── PhoenixFilament.Table.Cell (function component, per column)

PhoenixFilament.Pages.Resource.New  (LiveView — holds changeset state)
├── PhoenixFilament.Components.AdminLayout
│   └── <main slot>
│       └── PhoenixFilament.Form.Component  (function component)
│           ├── field components (text_input, select, etc. — all function components)
│           └── PhoenixFilament.Components.Button (submit)
```

### State Ownership

| State | Owner | How Updated |
|-------|-------|-------------|
| Authenticated user | Socket assigns (set in `on_mount`) | Read-only during session |
| Panel config (plugins, nav) | Socket assigns (set in `on_mount`) | Read-only during session |
| Table rows | Socket assigns | `handle_params` on navigate/filter/sort |
| Table filters/sort | Socket assigns | `handle_event` → `handle_params` |
| Form changeset | Socket assigns | `handle_event("validate")` |
| Flash messages | Socket assigns | After save/delete |

## Ecto Integration Points

Ecto schema introspection via `__schema__/1` enables auto-generation of sensible defaults:

- `schema.__schema__(:fields)` — list of all non-virtual fields
- `schema.__schema__(:type, field)` — infer default input type (`:string` → text_input, `:boolean` → checkbox, etc.)
- `schema.__schema__(:associations)` — detect belongs_to for auto-generating select options

This introspection runs at runtime (not compile time) to avoid coupling the framework's compile graph to every host app schema module. (HIGH confidence: `__schema__/1` is a standard Ecto runtime function, not a compile-time macro).

## Scalability Considerations

| Concern | Small app (1-10 resources) | Medium app (50+ resources) | Notes |
|---------|---------------------------|---------------------------|-------|
| Route generation | Flat route list in single scope | Group by plugin with nested scopes | Route count grows linearly with resources |
| Panel mount cost | Negligible | Register only active plugins | Lazy plugin registration by path prefix |
| Table queries | Direct Repo calls | Add Ecto query hooks for instrumentation | Framework should not own the query layer beyond pagination/filter/sort building |
| Component rendering | Full HEEx render each navigate | LiveView diff patching handles efficiently | No extra optimization needed — trust LiveView's diff engine |

## Build Order Implications for Roadmap

The architecture has strict bottom-up dependencies:

| Phase | What to Build | Why This Order |
|-------|--------------|----------------|
| 1 | Foundation + Component Library | No dependencies; unblocks everything else |
| 2 | Form Builder (standalone, no Panel) | Depends only on Component Library + Foundation |
| 3 | Table Builder (standalone, no Panel) | Depends only on Component Library + Foundation |
| 4 | Resource abstraction | Requires Form Builder + Table Builder to exist |
| 5 | Panel shell + routing + auth hook | Requires Resource to wire into LiveViews |
| 6 | Plugin architecture | Requires Panel to have a registration surface |
| 7 | Mix generator | Requires all above to exist to scaffold |

**Critical path:** Foundation → Form → Table → Resource → Panel → Plugin → Generator

Skipping ahead (e.g., building Panel before Table Builder exists) forces placeholder APIs that create rework. This order also matches FilamentPHP's actual package dependency graph (confirmed via DeepWiki source).
